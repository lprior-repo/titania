## Behavior Review

### Build Script (`build.rs`)

The build script performs two functions:

1. **`check_components()` (always runs):** Reads the local `rust-toolchain` TOML file and asserts that the configured components match an expected list (`["llvm-tools-preview", "rustc-dev"]`). No network access, no process spawning, no filesystem writes. Safe read + assertion only.

2. **`add_components()` (only under `#[cfg(docsrs)]`):** Under the `docsrs` cfg (injected exclusively by docs.rs during documentation builds), runs `rustup component add` for the two components. This only executes on docs.rs infrastructure, never in production builds.

**Verdict:** No build-script risk in normal builds. Docs.rs-only path is isolated and benign (invokes `rustup`, a trusted local tool).

### Proc-Macro / Code Generation

- **Zero proc-macro crates.** The crate uses `#![feature(rustc_private)]` to access internal rustc crates (`rustc_driver`, `rustc_data_structures`, `rustc_session`, `rustc_span`).
- All exported macros are `macro_rules!` invocations — compile-time text expansion only.
- The `dylint_library!` macro generates:
  - `extern crate rustc_driver` — links against `librustc_driver.so` (standard rustc plugin pattern)
  - A C-ABI function `dylint_version()` returning a version string (FFI stable symbol)
- No dynamic code generation, no runtime code emission.

### Unsafe Code

- **Zero `unsafe` blocks, zero `unsafe impl` blocks.** The only occurrences of the `unsafe` keyword are in `#[unsafe(no_mangle)]` attributes, which are stable Rust attributes (RFC 2972) for stable FFI symbol names — not an unsafe operation.

### File System Access

- **No direct file I/O in lib.rs.** The crate delegates all file operations to `dylint_internal::config`, which:
  1. Checks for `DYLINT_TOML` environment variable → if set, parses it as TOML from the env var string (no file read)
  2. Otherwise, runs `cargo_metadata::MetadataCommand` to find the workspace root, then reads `dylint.toml` from `workspace_root/dylint.toml`
  3. Parses the TOML into a `toml::value::Table` stored in a thread-safe `OnceLock`
- This is **read-only** access to a configuration file in the target project's workspace root. The path is determined by `cargo metadata` output (workspace root), not user input.
- **No writes, no appends, no deletes.** The config is read once at startup into a global `OnceLock`.

### Process / Network Access

- **No `std::process::Command` spawning** in lib.rs or build.rs (except docs.rs-only `rustup` call).
- **No network access.** No HTTP client, no socket operations, no DNS resolution.

### `rustc_private` Usage

- The crate links against `librustc_driver.so` and uses internal rustc crates directly via `extern crate`. This is the standard mechanism for Dylint libraries to integrate with the rustc plugin system.
- The `register_lints` function (generated by macros or implemented manually) receives a `&Session` and `&mut LintStore` — the standard rustc lint registration API.
- This is expected and correct behavior for a rustc plugin utility crate.

## Relationship to Titania

`dylint_linting` is a dependency of `crates/titania-dylint`, which is Titania's custom Dylint library. Titania uses two public APIs:

1. `dylint_linting::DYLINT_VERSION` — a constant string for FFI version reporting
2. `dylint_linting::init_config(sess)` — reads `dylint.toml` from the workspace root during rustc plugin initialization

The crate is **a tooling dependency only** — it is never compiled into production Titania binary artifacts. It only executes during `rustc` lint passes (via the `dylint-driver` tool) and reads a workspace config file.

## Safe-to-Deploy Decision

**APPROVE**

### Rationale

1. **No unsafe code.** Zero `unsafe` blocks or `unsafe impl` in the entire crate.
2. **No build-script risk.** The build script only reads a local TOML file and asserts on its contents; the only process spawning is docs.rs-only.
3. **No proc-macro code generation.** All macros are `macro_rules!` — simple compile-time text substitution with no dynamic evaluation.
4. **Limited filesystem access.** File I/O is read-only, targets a well-defined workspace root config file (`dylint.toml`), and is mediated through `dylint_internal::config` which is under the same authorship.
5. **No network access.** Zero network primitives, HTTP clients, or socket operations anywhere in the crate.
6. **No process spawning** in normal builds. Only docs.rs-only `rustup` invocation.
7. **Expected `rustc_private` usage.** Standard rustc plugin integration pattern, no exotic internal APIs.
8. **Clean dependency chain.** Dependencies are all well-established, widely used crates with strong provenance. `dylint_internal` is same-author and reviewed alongside.
9. **Tooling-only scope.** The crate only provides utilities for writing Dylint lint libraries — its runtime behavior is limited to config loading and FFI version reporting during rustc plugin initialization. It does not introduce any production runtime behavior into Titania.

### Caveats

1. `dylint_internal 6.0.1` has not been independently vetted in this report — it is reviewed alongside as same-author infrastructure. Its `config` module is the primary file-reading path; the audit of that module confirms it is read-only workspace config loading.
2. The crate uses `#![feature(rustc_private)]` which requires nightly Rust and links against unstable rustc internals. This is expected for Dylint libraries but means the crate can break on rustc upgrades. Not a security concern.
3. `cargo_metadata::MetadataCommand` invokes `cargo metadata` as a subprocess. This is a well-established, safe operation that only reads `Cargo.toml` files from the target workspace.
